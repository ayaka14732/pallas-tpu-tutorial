# 前言

本教程面向已经熟悉 JAX、希望进一步学习 TPU 自定义 kernel 开发的读者。你将从最基础的概念出发，逐步掌握在 TPU 上使用 Pallas 编写高性能自定义算子的能力。

## 为什么要学习 Pallas TPU Kernel 开发

Pallas 正在取代 JAX/XLA，成为 TPU 上性能关键 kernel 的直接表达层，并将重塑整个 TPU 算子生态。JAX 最核心的程序变换能力，如追踪（Tracing）、`vmap`、`jit` 仍然是上层系统的基础，但 Pallas 会替换的是围绕 Flax 等高层深度学习库、XLA 优化 pass 建立起来的旧生态。在新的生态里，kernel 不再只是编译器不可见的实现细节，而是模型研究者、编译器研究者和硬件设计者共同开发、组合、调优和演化的一等程序。

原因很简单：大模型时代，没有“能运行就够了”，只有充分优化。一次多余的 HBM 往返、一个无法融合的中间张量、一段没有重叠起来的通信与计算，乘上数万颗芯片和数月训练时间，都会变成真实而巨大的成本。XLA 必须服务广泛的模型，依靠通用分析和优化 pass 寻找解法；但新的 attention、ragged tensor、paged KV cache、稀疏路由、低精度格式以及跨芯片融合，往往需要算法、数据布局、内存层级和执行流水线共同设计，其中一些策略在高层 JAX/XLA 表达中根本无法完整描述。

与其等待某个模式被编译器识别、等待一个新的优化 pass 被设计和部署，不如由最了解模型与工作负载的人直接写出所需的 kernel。Pallas 给研究者的正是这一级控制权：决定如何切块，如何映射 program，数据以什么布局进入 VMEM，HBM 与 VMEM 之间的 DMA 如何同 MXU 计算重叠，何时同步，以及多个阶段如何组成软件流水线；同时，这些 kernel 仍然能够接受 JAX 程序变换，并进入统一的训练、推理与分布式执行系统。

掌握 Pallas，意味着能够把新的模型思想迅速变成高效实现。模型架构不会停在今天的 Transformer，硬件也不会停在某一代 TPU。库算子只能覆盖已经稳定下来的共性，而研究前沿恰恰发生在共性尚未形成的地方。理解 kernel 的分块、访存、累加、同步和布局，研究者就能在模型结构、数值格式或硬件约束变化时重新组织数据流，为训练和推理建立一条从算法原型到生产性能的短路径。

掌握 Pallas，是在参与 TPU 的软硬件协同设计。kernel 是模型需求与芯片结构相遇的地方：哪些数据流反复受制于带宽，哪些布局转换消耗过高，哪些同步原语缺失，怎样的片上存储、矩阵单元和互联才能支撑下一代工作负载，这些问题只有通过真实 kernel 才会暴露。今天对 Pallas kernel 的深入研究，不只是优化现有 TPU，也会为第 10 代乃至更远期 TPU 的指令、存储层级、互联与编程模型提供设计依据。

掌握 Pallas，是为自动调优的研发铺平道路。未来的 kernel 开发必然需要自动调优，而自动调优的研发与理解底层系统直接相关。搜索空间如何定义，tile、layout、pipeline stage 与并行策略如何参数化，哪些约束必须满足，代价模型观察什么信号，以及如何判断搜索结果是否真正逼近硬件上限，都建立在对 kernel 和 TPU 架构的深刻理解之上。Pallas 既是今天手工逼近峰值性能的工具，也是研发下一代 kernel 生成、代价模型和自动调优系统的实验场。

遗憾的是，有些用户给 Pallas 提出一些莫名其妙的 issue，根源往往只是想当然地要求 Pallas 像 JAX 一样工作。Pallas 不是另一套 JAX；它所表达的是 TPU 硬件已经具备的操作与执行能力，而不是凭空创造硬件并不支持的语义。系统学习 Pallas，正是为了正本清源，建立对其能力边界的基本认识，并尽可能减少这类由误解产生的无意义 issue。

因此，学习 Pallas 是为了获得贯通模型、编译器与芯片的能力：为今天的前沿模型释放 TPU 的全部性能，为明天的模型变化快速重构训练和推理系统，并把这些经验沉淀为未来自动优化与硬件设计的基础。

## 前置要求

本教程假定读者已经熟悉 Python 与 NumPy，能够阅读并编写常见的数组程序；同时熟悉 JAX 的基本使用方式，尤其理解 JAX 的追踪（Tracing）机制：JAX 如何记录 Python 函数中的数组计算，并将其交给 `jit`、`vmap` 等变换和编译流程处理。

读者应具备基础的深度学习背景，无须非常深入，但需要理解矩阵乘法、张量形状与常见算子的含义。

读者需要具备基础的 CUDA 和 GPU kernel 经验，无须非常深入，但至少应该能对一个简单的矩阵乘法 kernel 做基本优化。
