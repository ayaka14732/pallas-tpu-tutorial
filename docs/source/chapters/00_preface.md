# 前言

本教程面向已经熟悉 JAX、希望进一步学习 TPU 自定义 kernel 开发的读者。你将从最基础的概念出发，逐步掌握在 TPU 上使用 Pallas 编写高性能自定义算子的能力。

## 为什么要学习 Pallas TPU Kernel 开发

在传统的 JAX 程序中，我们并不需要手写 kernel。`jax.numpy`、`jax.lax`、XLA 编译器以及上层深度学习框架已经覆盖了大量常见计算：矩阵乘法、卷积、归一化、逐元素运算、规约、自动微分和跨设备并行。对于这些标准路径，最好的做法通常是写出清晰的 JAX 程序，让编译器完成融合、布局选择和底层代码生成。

但是，当模型结构、数据形状或性能目标开始偏离标准算子时，高层抽象就会逐渐显露边界。一个看似简单的算子，如果会产生巨大的中间张量，可能受限于 HBM 带宽而不是算力；一个 attention 变体，如果需要特殊的分块、掩码或缓存布局，可能无法直接由现有库函数表达；一个 ragged、paged 或 sparse 的工作负载，如果访问模式不规则，可能需要开发者显式安排数据搬运和计算顺序。此时，继续堆叠高层算子往往只能得到“语义正确但性能不理想”的程序。

Pallas 的价值正在于填补这一层空白。它允许我们在 JAX 程序内部为关键路径编写自定义 kernel，同时保留 JAX 的数组模型、编译流程和整体工程形态。换句话说，学习 Pallas 就是要学习在抽象不够用的时候，如何把少数最重要的计算段落下降到更接近硬件的数据流描述。

对于 TPU 来说，这一点尤其重要。TPU 的执行模型和 GPU 并不相同：计算主要围绕 MXU、VMEM、SMEM、DMA 和软件流水线展开；TPU kernel 的性能不只取决于“并行线程够不够多”，还取决于数据是否以合适的块形状进入 VMEM，HBM 到 VMEM 的传输是否能够和 MXU 计算重叠，grid 的顺序是否有利于数据复用，以及布局是否适合向量寄存器和矩阵单元。Pallas 暴露的正是这些关键控制面：如何切块，如何映射 program，如何选择内存空间，如何发起异步拷贝，如何组织流水线。

因此，学习 Pallas TPU kernel 开发至少有几类实际收益。

第一，它能帮助我们理解高性能算子背后的结构。RMSNorm、Softmax、矩阵乘法、FlashAttention、Paged Attention 等算子的背后是一系列关于分块、访存、累加、同步和布局的工程决策。理解这些决策之后，读者再回头看 JAX、XLA 或深度学习框架中的算子行为，会更容易判断性能瓶颈究竟来自算力、带宽、布局转换，还是不必要的中间结果。

第二，它让我们有能力处理非标准模型带来的性能问题。研究和生产系统中经常会出现库函数没有完全覆盖的场景，例如新的 attention 变体、特殊 mask、分块缓存、稀疏路由、长上下文推理中的分页 KV cache，或多个小算子的定制融合。Pallas 提供了一条仍然留在 JAX 生态内的实现路径，使这些实验不必一开始就落到完全独立的底层运行时中。

第三，它能训练一种可迁移的性能直觉。即使最终并不手写很多 kernel，理解 TPU 上的内存层级、数据搬运和流水线，也会改变我们编写高层 JAX 代码的方式。读者会更自然地关注张量形状是否稳定、维度顺序是否合理、是否创建了过大的临时数组、是否把可融合的计算拆散、是否让编译器难以看见真实的数据流。这些判断不仅适用于 Pallas，也适用于使用 JAX 调优大模型训练和推理的日常工作。

## 前置要求

本教程假定读者已经熟悉 Python 与 NumPy，能够阅读并编写常见的数组程序；同时熟悉 JAX 的基本使用方式，尤其理解 JAX 的追踪（Tracing）机制：JAX 如何记录 Python 函数中的数组计算，并将其交给 `jit`、`vmap` 等变换和编译流程处理。

读者应具备基础的深度学习背景，无须非常深入，但需要理解矩阵乘法、张量形状与常见算子的含义。

读者需要具备基础的 CUDA 和 GPU kernel 经验，无须非常深入，但至少应该能对一个简单的矩阵乘法 kernel 做基本优化。
