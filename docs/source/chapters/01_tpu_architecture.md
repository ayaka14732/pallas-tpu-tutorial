# 第 1 章：TPU 硬件架构与执行模型

在开始编写 Pallas Kernel 之前，理解 TPU 的硬件架构是至关重要的。许多在 GPU 上习以为常的编程模式，在 TPU 上可能会导致严重的性能下降。本章将带你深入了解 TPU 的核心组件、内存层级，以及它与 GPU 的本质区别。我们将探讨硬件特性如何直接影响我们编写算子的方式。

## TPU vs GPU：本质的架构差异

最重要的一点认知是：**TPU 并不是 GPU。** 它们在设计哲学上有着根本的分歧。

### GPU (如 NVIDIA A100/H100)：大规模并行机器
GPU 是高度并行的机器，拥有成千上万个计算核心（CUDA Cores）。在 CUDA 编程模型中，我们通过海量的线程（Threads）和线程块（Blocks）的并发执行来隐藏内存延迟。当一个线程束（Warp）等待内存加载时，调度器会立即切换到另一个准备好计算的线程束。这种架构非常适合处理极其零散、分支密集的并行任务。

### TPU：超宽向量寄存器的顺序机器
相反，**TPU 是一台拥有超宽向量寄存器的顺序执行机器**。在执行模型上，它其实**更像是一个拥有巨大 SIMD（单指令多数据流）单元的单核 CPU**。

TPU 并没有成千上万个并发线程。它通过**软件流水线（Software Pipelining）**和**异步执行（Asynchronous Execution）**来隐藏延迟。这意味着在 TPU 上，作为程序员（或编译器），你必须显式地安排内存搬运和计算的重叠，而不是依赖硬件调度器。

在 TPU 上，软件可以安排某些操作在后台执行，使它们与主指令流异步进行。这包括：
- **DMA 传输**：HBM（高带宽内存）的访问不能由计算指令直接发起，必须由专门的 DMA 单元预取到较低层级的内存（VMEM）。
- **矩阵乘法**：由专门的矩阵乘法单元（MXU，Matrix Multiply Unit）支持。MXU 是一个巨大的脉动阵列（Systolic Array），在执行大块矩阵乘法时效率极高。
- **矩阵转置和排列**：由专门的 XLU（Cross-Lane Unit）单元支持。

## 内存层级（Memory Hierarchy）

TPU 拥有自己独特的内存层级。在 CUDA 中，我们经常讨论全局内存（Global Memory）、共享内存（Shared Memory）和寄存器（Registers）。在 TPU 中，概念类似，但容量和使用方式大不相同。

| 内存空间 | 全称 | 容量示例 (v4/v5) | 对应 CUDA 概念 | 特点与用途 |
| :--- | :--- | :--- | :--- | :--- |
| **HBM** | High Bandwidth Memory | 16GB - 32GB | Global Memory | 主内存。延迟极高。Kernel **不能**直接在 HBM 上进行计算，必须先将数据搬运到 VMEM。 |
| **VMEM** | Vector Memory | 16MB+ | Shared Memory | 向量内存。容量比 GPU 的共享内存（通常 160KB 左右）**大得多**。所有向量计算的输入和输出都必须在这里。与 HBM 的通信以 4KiB 为粒度。 |
| **SMEM** | Scalar Memory | 几百 KB | Constant Memory / Shared | 标量内存。附属于标量核心，支持随机访问。只需单条指令即可读写 32-bit 值。通常用于存放控制流数据、动态索引（如 Sparse Kernel）。 |
| **VREG** | Vector Registers | - | Registers | 向量寄存器。TPU 执行大部分计算的地方。对于 32-bit 值，大小通常为 8x128。 |

**优化启示：** TPU 的 VMEM 非常大（16MB 级别），这意味着在 TPU 上，我们可以使用比 GPU 大得多的块（Block Sizes）。在编写 FlashAttention 时，这允许我们一次性处理更长的序列片段，从而减少 HBM 读写次数。

## 数组布局与分块（Tiling）

在高级 JAX 程序中（如 `jax.jit` 内部），中间数组的维度顺序通常不影响性能，因为编译器可以自由重排它们。但在 Pallas 中，**维度顺序对生成的代码质量有巨大影响**。

TPU 在 2D 向量寄存器上执行大部分计算。以 TPU v6 为例，32-bit 值的寄存器大小为 **8x128**。

当一个向量值从 VMEM 加载到寄存器时（例如 `x = x_ref[...]`），数组的**最后两个维度**将被平铺（Tiled）到寄存器中。Pallas 只会考虑将中间数组的最后两个维度映射到 8x128 向量寄存器的维度（分别称为 sublanes 和 lanes）。

### 布局带来的深远影响：

1. **最后两个维度的特殊性**：涉及最后两个维度的归约（Reduction）、重塑（Reshape）和转置（Transpose）通常比其他维度更昂贵。有些涉及最后两个维度的 Reshape 是不支持的。
2. **避免单元素维度**：在最后两个轴上保留单元素维度（Singleton dimensions，如大小为 1 的维度）是非常浪费的。这会消耗过多的寄存器，可能导致寄存器溢出（Spill）到 VMEM，从而导致性能断崖式下跌。
3. **自动填充（Padding）代价高昂**：所有向量计算都会向上填充到 Tile 大小。两个 `1x1` 数组相加的代价与两个 `8x128` 数组相加一样大。而两个 `8x128x1x1` 数组相加的代价是 `8x128` 数组的 1024 倍，因为它会被填充为 `8x128x8x128`。

**优化启示：** 在设计 Kernel 时，始终确保计算密集的维度（特别是参与矩阵乘法的维度）位于张量的最后两维，并且大小最好是 128 和 8 的倍数。

## 计算分配（Compute Placement）

TPU 包含两种核心：
- **向量核心（Vector Core）**：处理 1D 及以上的数组操作。
- **标量核心（Scalar Core）**：处理 0D（标量）数组操作。

所有标量（即 0D）数组都将存储在标量寄存器中，对它们的操作将在标量核心上执行。所有其他操作（即使是单元素但维度大于等于 1D 的数组）都将在向量核心上执行。

## 网格执行顺序（Grid Execution Order）

由于 TPU 是高度顺序化的机器，Pallas 中的网格（Grid）通常**不是并行处理的，而是按字典序（Lexicographic order）顺序执行的**。

这解锁了一些在 GPU 上完全不可思议的特性：
1. **自动 HBM 复用**：当两个连续的网格索引使用相同的输入切片时，第二次迭代的 HBM 传输会被编译器自动跳过，因为数据已经存在于 VMEM 中。
2. **无竞态条件的安全写入**：Kernel 的多次调用可以写入输出的同一个切片，而没有任何竞态条件（Race Condition）的风险。在 GPU 上，多个 Block 写入同一位置需要复杂的原子操作（Atomic Ops），而在 TPU 上，只要所有写入特定切片的调用是连续的，就可以直接覆盖写入。

*注：在较新的 TPU 代次中，一个芯片上的两个核心（Megacore）通常被抽象为单个设备。Pallas 可以打破顺序执行保证，将网格的一个轴在核心上并行化，这被称为 `dimension_semantics=["parallel", ...]`。我们将在进阶章节讨论。*
